# Generator Schema V2 Notes

The current `generator.md` schema is strongest for fixed finite trajectory fields:

```text
X: D -> A
c = [t,x,y,z]
```

with frontier-based writes and rule-based reads. To support the broader set of systems from the NKS index, V2 should keep that model as the fixed-lattice case, but add a more general layer above it.

## Main Change

Generalize from:

```text
X: D -> A
D subset Z^4
RULE returns one value in A
```

to:

```text
STATE_k = (CARRIER_k, VALUES_k, TOPOLOGY_k?)
RULE returns a PATCH
STATE_{k+1} = APPLY(STATE_k, PATCHES)
```

Here `CARRIER` can be a grid, string, graph, object set, branch set, or continuous field.

## Specific Modifications

### 1. Add `CARRIER_TYPE`

Examples:

```text
LATTICE
SEQUENCE
GRAPH
TREE_EXPRESSION
GEOMETRIC_OBJECT_SET
MULTIWAY_STATE_GRAPH
CONSTRAINT_DOMAIN
CONTINUOUS_FIELD
```

For ordinary cellular automata, `CARRIER_TYPE = LATTICE` and the current schema works almost unchanged.

### 2. Allow Dynamic Domains

Current `D` is fixed. Substitution systems, tag systems, networks, and geometric replacement systems need domains that grow, shrink, or rewire.

Add:

```text
DOMAIN_MODE in {FIXED, DYNAMIC}
```

and allow:

```text
C_k = carrier at construction round k
X_k: C_k -> A
```

### 3. Make Rules Return Patches

Current rules return a new value for one coordinate. General systems need structural edits.

Patch operations might include:

```text
ASSIGN_VALUE
INSERT_ELEMENT
DELETE_ELEMENT
REPLACE_BLOCK
APPEND_BLOCK
SPLIT_OBJECT
ADD_NODE
DELETE_NODE
ADD_EDGE
REMOVE_EDGE
REWIRE_EDGE
REWRITE_SUBTREE
EMIT_BRANCH
ASSERT_CONSTRAINT
```

This would cover substitutions, tag systems, graph rewrites, symbolic systems, and multiway systems much more naturally.

### 4. Generalize `FRONTIER` To `MATCHES`

For cellular automata, the frontier is writable coordinates. For other systems, the active units are substrings, graph neighborhoods, expression subtrees, templates, or constraint regions.

Replace or extend:

```text
FRONTIER
```

with:

```text
MATCH_SELECTOR
```

Examples:

```text
all cells in next slice
first matching substring
all matching substrings
all graph nodes matching local topology
all expression subtrees matching pattern
all unsatisfied constraint regions
```

### 5. Add `SCHEDULE`

The current schema assumes parallel writes. Many systems need different update schedules.

```text
SCHEDULE in {
  PARALLEL_ALL,
  SEQUENTIAL_FIRST_MATCH,
  SEQUENTIAL_SCAN,
  ACTIVE_HEAD,
  ACTIVE_SET,
  ALL_POSSIBLE_REWRITES,
  SOLVER,
  CONTINUOUS_TIME
}
```

This is probably the most important addition after dynamic domains.

### 6. Add Branching Semantics For Multiway Systems

Multiway systems do not produce one next state; they produce many.

Add:

```text
EVOLUTION_MODE in {SINGLE_TRAJECTORY, MULTIWAY}
```

For multiway systems:

```text
STATE_GRAPH = (states, transitions)
```

instead of a single trajectory field.

### 7. Add Constraint/Solver Mode

Constraint systems are not generated by forward evolution. They need a solver interface.

```text
SYSTEM_MODE in {EVOLUTION, CONSTRAINT_SATISFACTION}
```

Constraint fields:

```text
CONSTRAINTS
LOCAL_TEMPLATES
REQUIRED_TEMPLATES
SOLVER
SEARCH_DOMAIN
SATISFIABILITY_POLICY
```

### 8. Add Numeric/Continuous Semantics

For iterated maps, continuous cellular automata, and PDEs:

```text
VALUE_SPACE in {FINITE_ALPHABET, INTEGER, RATIONAL, REAL, COMPLEX, FIELD}
PRECISION_POLICY
SOLVER
DISCRETIZATION
```

PDEs especially need an explicit distinction between the mathematical rule and the numerical approximation.

## Generator Family

V2 should become a family of generators:

```text
GridTrajectoryGenerator        current schema
SequenceRewriteGenerator       substitution/tag systems
GraphRewriteGenerator          network systems
ExpressionRewriteGenerator     symbolic systems
MultiwayGenerator              branching rewrite systems
ConstraintGenerator            template/local constraint systems
NumericMapGenerator            arithmetic and iterated maps
ContinuousFieldGenerator       continuous CA and PDEs
```

Shared concepts:

```text
STATE
SEED
MATCH_SELECTOR
RULE
SCHEDULE
BOUNDARY
PATCH
RENDERING
```

This preserves the current cellular automata schema instead of overloading it, while making the adjacent systems fit natively.
